1. Executive summary

Axeous Learn currently supports two kinds of plans: standard plans seeded into the catalog by the dev team, and custom plans that end users build from documents they upload. Standard plans are authoritative but hard to scale (every content edit is a deploy); custom plans scale infinitely but carry no quality guarantee.

The Official Plan Program closes the gap. It introduces a curated, subject-matter-expert-reviewed content production pipeline whose output ships as official plans — durable, versioned, paid products that subscribers can trust. Reference materials live in the repository under version control. Question generation, expert audit, and release management live in the application database, surfaced through new admin and contributor interfaces. A new contributor role lets vetted experts (dentists, dental students, etc. for DAT; equivalents for other exams) review AI-generated questions for a per-review fee, with monthly payouts.

The pilot is the DAT, for which authoritative reference material already exists in the repository (reference/dat/Mock_DAT_Tutor_Edition.md) and the canonical topic structure is well-defined. Subscriber access is gated by monthly Stripe subscriptions, configured per exam.

2. Background

Axeous Learn ships exam prep across multiple high-stakes credentialing exams. The platform today seeds exam metadata in server/src/config/exams.ts and stores all questions, plans, and user activity in MongoDB. Question generation is powered by Anthropic Claude (Haiku 4.5 for topic extraction, Sonnet 4.6 for question synthesis), already wired into the custom-plan flow.

Two facts about today's system motivate this proposal:

  1. Generation infrastructure already exists, but is locked to custom plans. The same pipeline that turns a user's uploaded PDF into topics and questions could produce official content if pointed at a curated corpus and reviewed by experts.
  2. The only quality signal today is user complaint. Questions surface in test sessions immediately; quality issues are caught reactively via the reportCount >= 5 flag mechanism. There is no proactive review queue and no way for a domain expert to validate content before it reaches paying customers.

The Official Plan Program builds on existing infrastructure rather than replacing it. Custom plans continue to work as they do; standard exam seeding continues to work as it does; the new program adds a third content path optimized for paid, expert-validated content.

3. Problem statement

Axeous Learn needs a way to:

  • Produce large volumes of high-quality, exam-accurate questions for credentialing exams without expanding the engineering team.
  • Verify content quality through subject-matter experts who are not employees and do not work in the repository.
  • Bundle that content into named, versioned releases that can be marketed and sold.
  • Gate access to those releases via subscription, with clean rollback if a release has issues.
  • Pay contributors transparently and resist gaming of the review workflow.

None of these are solved by the current standard or custom plan flows.

4. Goals & non-goals

Goals

  • Ship a first Official Plan for the DAT, generated from the existing reference corpus and reviewed by paid SMEs.
  • Establish a repeatable program model that generalizes to NCLEX, MCAT, USMLE, LSAT, CPA, and bar exam content.
  • Provide a contributor experience that a working clinician can use in 15-minute windows between patients.
  • Provide an admin experience that allows non-engineers to manage releases, payouts, and contributor scope.
  • Resist obvious gaming patterns (drive-by approvals, duplicate accounts, dwell-time evasion) without making the contributor flow feel punitive.

Non-goals

  • Public contributor sign-up. Contributor onboarding is invite-only.
  • Real-time collaborative editing of reference content. Reference material is edited via git PRs.
  • Vector database for retrieval. The existing chunked-text + Claude context-window approach is sufficient at our current corpus sizes; revisit if a single exam's reference corpus exceeds ~50 MB.
  • Replacing the standard or custom plan paths. They continue to exist and serve their respective audiences.
  • Self-serve contributor payout (Stripe Connect, 1099 automation). v1 uses a CSV export to accounting; revisit once payout volume justifies the integration cost.

5. Solution overview

The Official Plan Program is a hybrid content system:

  • Reference material lives in the repository under reference/<exam-code>/, with a per-exam manifest.yaml and per-file frontmatter. This material is edited via git pull requests and is authoritative.
  • Generated content lives in the database. The existing Claude-based generation pipeline is extended to stamp each question with the corpus version it was generated against, plus stable citation handles (file hash + chunk index) into the reference material.
  • Expert review lives in the database, surfaced through a new contributor UI. Reviewers approve or reject questions; votes are immutable events that drive both content state and per-review billing.
  • Releases bundle approved questions into named, versioned units ("DAT Summer 2026") that admins promote to live. Subscribers see questions only from currently-live releases for exams they have purchased.
Why hybrid Repo-side gives us version control, PR review, and a single source of truth for the canonical corpus. DB-side gives us a workflow that non-engineers can use, a real audit trail of expert decisions, and the ability to ship content without redeploying. Each side gets what it's good at.

6. Pilot: DAT

The DAT is the pilot exam because:

  • The reference corpus already exists at reference/dat/Mock_DAT_Tutor_Edition.md (a tutor-edition mock exam authored for the platform).
  • The DAT topic structure is well-defined and stable: four sections (Survey of Natural Sciences, Perceptual Ability Test, Reading Comprehension, Quantitative Reasoning) with documented subsections and question counts.
  • The dental community (practicing dentists, dental students, faculty) is a tractable contributor pool with strong domain expertise and visible credentials.
  • It is not currently in the seeded exam list, so launching it as a paid product avoids disrupting any existing free users.

The DAT pilot will validate the program model end-to-end: corpus loading, generation, contributor invite, review queue, release publishing, subscription checkout. Subsequent exams will reuse the same machinery with different reference corpora and contributor pools.

Contributor scope is per-exam: a contributor invited for DAT can review only DAT questions, even if other exams are later added. A single contributor can be scoped to multiple exams, each with its own per-review rate.

8. Contributor program

Onboarding

Contributor onboarding is invite-only. Admin generates an invite token bound to an email address, a list of exam scopes, and per-exam rates. The invitee signs up through standard auth (Google OAuth via Better Auth), is upgraded to the contributor role, and lands on the review queue on next login.

Compensation

Contributors are paid per billable review, with rates configured per (contributor, exam) pair. This lets us pay a board-certified specialist more than a final-year student, and reflect domain difficulty (DAT review may pay differently than CPA review).

A review is "billable" if and only if:

  • The reviewer's role and scope permit reviewing the question.
  • Dwell time on the question meets the minimum threshold (default: 5 seconds).
  • The reviewer has not previously voted on this question.
  • A rejection vote includes a non-empty comment.
  • The reviewer has not exceeded the configured daily cap (default: 200 billable reviews per day).
  • The review has not been retroactively flagged by admin spot-audit.

Skipped questions, duplicate votes, and below-threshold dwell votes are recorded for audit but do not count toward earnings.

Payout

Payouts run on a monthly cycle. At month end, a job aggregates billable reviews per contributor, applies their per-exam rate, and creates a ContributorPayout record. Admins review the period, mark payouts paid (with an external reference: wire ID, check number, etc.), and export a CSV for accounting. Reversals are supported for retroactive flags.

Quality signals

Each contributor accumulates a reliability score: the percentage of their approvals that are later overturned (rejected by another reviewer, removed via the flagged-question system, or rejected in admin spot-audit). Low scores trigger an admin audit prompt — they do not automatically suspend the contributor. The intent is to surface problems for human judgment, not to police reviewers algorithmically.

9. Content lifecycle

Generated questions move through five states:

StateMeaningVisible to
draftJust generated; not yet eligible for reviewAdmin only
pendingIn the review queue, awaiting contributor votesAdmin + contributors with matching scope
approvedMet the approval threshold; eligible for inclusion in a releaseAdmin + contributors
rejectedMet the rejection threshold; preserved for audit and prompt-engineering feedbackAdmin only
publishedIncluded in a live release; served to subscribersEveryone (rendered in tests)

Approval and rejection thresholds (default: 2 approvals with 0 rejections, or 2 rejections) are configurable. Mixed votes (one approve, one reject) escalate to admin.

The split between approved and published is deliberate. Approved questions are quality-cleared but not yet released — they wait in inventory until an admin includes them in a named release. This lets us ship content in coherent batches with marketing copy, and lets us roll back a release without touching question state.

10. Subscription model

Official plans are gated by monthly Stripe subscriptions, configured per exam. Each exam has its own Stripe product. The flow:

  1. A signed-in user views a gated exam in the catalog. The exam shows a lock and a "Subscribe" CTA.
  2. Checkout creates a Stripe subscription. On checkout.session.completed, our webhook appends the exam code to user.examAccess.
  3. Subsequent test session, plan creation, and question fetch requests for that exam are gated by a requireExamAccess(examCode) middleware.
  4. On subscription cancellation or payment failure, the webhook removes the exam code from examAccess. The user retains their progress and history; they just can't start new sessions until they resubscribe.

Admins can grant examAccess manually for comp'd accounts, support cases, and contributor courtesy access. The custom-plan flow is unaffected — users who build their own plans continue to do so under their existing license entitlements.

11. Risks & mitigations

RiskMitigation
Contributors game the per-review payment model (drive-by approvals, sock puppets, dwell-time evasion). Min dwell time, daily cap, comment-required rejections, duplicate-vote guard, reliability scoring, admin spot-audit with retroactive billable flag.
Approved content turns out to be wrong; subscribers see bad questions. 2-of-2 approval default; reliability scoring on reviewers; question reports continue to work post-publish; release-level rollback flips bad content out without touching question state.
Reviewer churn — contributors get bored, throughput drops. Keyboard-driven queue UI optimized for short sessions; clear per-period earnings dashboard; allow multiple exam scopes so contributors can switch subject matter.
Stripe webhook drops or arrives out of order; users lose access incorrectly. Idempotent webhook handler keyed on Stripe event ID; nightly reconciliation job that syncs examAccess against Stripe's subscription state of record.
Reference corpus edit invalidates existing citations on already-published questions. Corpus versioning: each release pins to a corpusVersion; published questions retain the citation hashes they were generated against; new generations target the new version.
Contributor scope sprawl as exams are added; admin overhead grows. Bulk-invite and bulk-rate UI on day one; contributor self-service for profile and scope visibility; no admin involvement in routine review work.

12. Success metrics

The pilot is successful if, six months after DAT launch:

  • Content velocity — at least one named release shipped per month, each containing ≥ 100 published questions across DAT sections.
  • Reviewer engagement — at least 5 active contributors with ≥ 50 billable reviews per month each.
  • Content quality — published question report rate (user reports per 100 served impressions) below the baseline observed on standard-plan content.
  • Commercial signal — subscription conversion rate from gated catalog view, and month-2 retention on DAT subscribers, both within the range we'd want to scale.
  • Operational cost — contributor payout as a percentage of subscription revenue is sustainable at our target gross margin.
