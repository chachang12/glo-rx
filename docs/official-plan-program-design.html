1. Overview

This document specifies the engineering work required to deliver the Official Plan Program. It defines repo conventions for reference material, the database schema changes, the question lifecycle state machine, the contributor and admin API surfaces, the release model, and the Stripe subscription integration. The DAT is the pilot exam, but every component is designed to generalize across exams.

The high-level shape: reference material remains under git; generated content, expert votes, releases, payouts, and subscription state live in MongoDB; the React client gains two new role-scoped surfaces (admin extensions and a contributor app); Stripe gates per-exam access via webhooks.

2. System architecture

Boundaries

ConcernLives inWhy
Canonical reference material Git, under reference/<exam-code>/ Authoritative; needs PR review, diff-able history, content-addressed citation stability.
Loaded reference chunks MongoDB (PlanDocumentModel, extended) Generation pipeline needs random access; chunk hashes anchor citations.
Generated questions MongoDB (QuestionBankModel) Production at scale; lifecycle state changes; per-user exposure tracking.
Review votes & billing events MongoDB (ReviewEvent, new) Immutable per-event records; audit and payout source of truth.
Releases MongoDB (Release, new) Operational artifact managed by non-engineers; needs rollback.
Subscription & access MongoDB (user.examAccess) mirrored from Stripe Stripe is source of truth for billing; we mirror entitlement for query speed.

Data flow

repo (reference/dat/*.md, manifest.yaml)
  → loader (sha256 per file, chunked into PlanDocumentModel with corpusVersion)
  → admin generation console
  → Claude Sonnet 4.6 (via existing generate-questions.service.ts)
  → QuestionBankModel (status: 'draft', stamped with corpusVersion + citation hashes)
  → admin promotes batch to 'pending'
  → contributor review queue (ReviewEvent records per vote, billable flag computed)
  → 2 approvals → status: 'approved'  |  2 rejections → status: 'rejected'
  → admin assembles approved questions into a Release (status: 'draft')
  → admin publishes Release (status: 'live'); included questions → 'published'
  → test sessions for subscribed users serve questions where releaseId ∈ live releases

Manifest

reference/<exam>/manifest.yaml is the loader's entry point:

exam: dat
corpusVersion: 2026.06.01     # bump on any content change
files:
  - path: biology/cell-biology.md
    role: reference
  - path: biology/genetics.md
    role: reference
  - path: pat/apertures.md
    role: reference
  - path: official-tests/mock-dat-tutor.md
    role: official-test       # parsed by a different loader path
generation:
  defaultTypeWeights:
    mcq: 1.0                  # DAT is MCQ-only on the real exam
  defaultDifficulty: medium

Loader behavior

A new server-side script server/src/scripts/load-corpus.ts handles ingestion. Invocation: npm run load-corpus -- --exam dat.

  1. Read reference/<exam>/manifest.yaml; resolve corpusVersion.
  2. For each role: reference file: read, validate frontmatter, compute sha256(fileBytes), chunk text (existing chunker from custom-plan pipeline), upsert into PlanDocumentModel keyed by (examCode, corpusVersion, fileHash).
  3. For each role: official-test file: parse using the existing official-test importer; upsert into OfficialTestModel.
  4. Write a CorpusVersion registry document recording the version, file list, hashes, and load timestamp. Subsequent loads with the same version are idempotent; a version bump triggers fresh inserts side-by-side with old chunks (old chunks are retained because they may still be cited by existing questions).

The loader does not mutate or delete prior corpus versions. Garbage collection of unreferenced chunks is a manual admin action, not a load-time side effect.

{
  reviewerId: ObjectId,
  questionId: ObjectId,
  examCode: string,

  comment: string | null,

Vote revisions are handled by inserting a new event with billable: false and notBillableReason: 'duplicate', alongside an update to the prior event's vote field. The prior billable event stays billable. (Alternative considered: rewrite the prior event. Rejected because it breaks the immutability invariant that makes the payout aggregation trustworthy.)

5. Question lifecycle

draft → pending → approved → published ↘ rejected
TransitionTriggerGuard
(new) → draftGeneration completesQuestion passes validation tool-use schema
draft → pendingAdmin clicks "Promote to review"Admin role; bulk-allowed
pending → approved2nd approval vote without prior rejectionDefault threshold; admin-configurable per exam
pending → rejected2nd rejection voteEach rejection must include a comment
pending → admin escalationMixed votes (1 approve, 1 reject)Surfaced on admin dashboard, not a status change
approved → publishedIncluded release transitions to liveRelease publish action; sets releaseId
published → approvedRelease archivedRollback; question becomes inventory again
any → rejectedAdmin manual rejectAdmin role; preserves audit trail

Only published questions are served to test sessions. approved questions are inventory.

6. Voting & anti-gaming

Vote submission flow

  1. Contributor views a question; client starts a dwell timer on first render.
  2. Contributor presses A / R / S (or clicks button).
  3. Client POSTs { questionId, vote, comment?, dwellMs } to /api/contributor/review.
  4. Server validates: role + scope, no prior event for (reviewer, question), comment present on reject, dwell ≥ threshold, daily cap not exceeded.
  5. Server inserts ReviewEvent with computed billable and notBillableReason.
  6. If event is recorded (regardless of billable), server updates QuestionBankModel.votes, recomputes approvalCount / rejectionCount, and transitions status if threshold met.
  7. Server responds with next question in the queue.

Billable determination

ConditionEffect
dwellMs < minDwellMs (default 5000)Vote recorded, billable: false, reason below-dwell; status update still applies (we count the vote, we just don't pay for it).
Reviewer's billable count this UTC day >= dailyCapVote recorded, billable: false, reason over-cap.
Vote on a question the reviewer has already voted onInsert rejected (unique index violation); admin path exists to revise.
Admin spot-audit flags after the factbillable set to false, reason admin-flagged; next payout shows a negative adjustment line.
Skip (vote == null in UI)No ReviewEvent created at all; question is hidden from this reviewer's queue for the session.
Design decision Below-dwell votes still count toward question state transitions. The reasoning: if a reviewer can correctly approve a question in 3 seconds, the vote is probably right; we just don't want to pay for that level of effort. This may be wrong — if we see below-dwell votes correlating with bad outcomes, switch to "below-dwell votes recorded but do not affect state."

Reliability score

Nightly job:

reliability = 1 - (overturned_approvals / total_approvals)

overturned = approval where the question was later:
  - rejected by another reviewer, OR
  - removed via flagged-question admin action, OR
  - rejected in admin spot-audit, OR
  - reported by users at >= 5x baseline rate post-publish

Score is denormalized to user.contributor.reliabilityScore for sort. No automated action is taken on low scores; admin sees them on the Contributors screen and decides.

7. Release & versioning

Versioning scheme

Releases use a calendar-based version per exam: YYYY.MM, optionally suffixed with .N for multiple releases in a month. Examples: 2026.06, 2026.06.2, 2026.07. The version is decoupled from the corpus version; a single release may pull from one corpus version, but a corpus version may produce many releases.

Release composition flow

  1. Admin opens the Releases screen for an exam; clicks "New release".
  2. Admin picks from approved questions, filtered by topic / corpus version / date approved.
  3. Admin sets name, version, and notes. Saves as status: 'draft'.
  4. Admin previews (renders sample of questions as they will appear).
  5. Admin clicks "Publish". Server:
    • Validates no other release for this exam is in mid-transition.
    • Bulk-updates included questions to status: 'published', sets releaseId and publishedAt.
    • Updates the release to status: 'live'.

Multiple live releases per exam

An exam can have multiple live releases simultaneously. Test sessions query { examCode, status: 'published', releaseId: { $in: liveReleaseIds } }. This lets us ship incremental releases without retiring prior content.

Rollback & archive

"Archive" a release: set status: 'archived', set archivedAt / archivedBy. The included questions revert to status: 'approved' and have releaseId cleared. They become inventory again. Subscribers stop seeing those questions immediately.

Archive is the rollback mechanism. There is no "edit a live release in place" path — to swap content, archive the current release and publish a new one.

10. Stripe integration

Products

One Stripe Product per exam, with one monthly Price. Naming convention: product_<examCode> in metadata, "Axeous Learn — DAT" as display name. Product IDs are stored in a server-side config keyed by exam code (not in the database — they're deployment configuration).

Checkout flow

  1. User clicks "Subscribe" on a gated exam.
  2. Client POSTs /api/billing/checkout-session with { examCode }.
  3. Server creates a Stripe Checkout Session with the matching Price, customer_email from the authed user, and client_reference_id = userId. Returns the session URL.
  4. Client redirects to Stripe Checkout.
  5. On success, Stripe redirects to /billing/success?session_id={CHECKOUT_SESSION_ID}. Webhook does the entitlement update (do not rely on the redirect).

Webhook events

EventAction
checkout.session.completedResolve user from client_reference_id; resolve examCode from product metadata; upsert examAccess entry with source: 'stripe' and expiresAt = subscription.current_period_end.
customer.subscription.updatedRefresh expiresAt from current_period_end.
customer.subscription.deletedRemove the matching examAccess entry.
invoice.payment_failedNo immediate access change; Stripe retries; eventual subscription.deleted handles termination.

Idempotency & reconciliation

Webhook handler is idempotent on event.id via a ProcessedStripeEvent collection (insert with unique constraint; if duplicate, ack and skip). A nightly reconciliation job lists all active Stripe subscriptions, diffs against user.examAccess entries where source: 'stripe', and reconciles any drift.

12. Migration plan

Schema migration

All new fields on User and QuestionBankModel are nullable / default-valued, so Mongoose handles them implicitly on next write. The only explicit migration needed:

  • Backfill existing QuestionBankModel documents with status: 'published' so test sessions continue to serve them. Set releaseId to a synthetic "legacy" release per exam (created during migration) so the multi-live-release query model works uniformly.
  • Backfill existing users with role: 'user' (already the default), examAccess: [], contributor: null.

Migration script: server/src/scripts/migrate-official-plan-program.ts. Idempotent — checks for the legacy release before creating one.

Phase ordering (mirrors the proposal phases)

  1. Phase 0: Schemas + migration + DAT exam seed + topic seed + corpus loader.
  2. Phase 1: Admin generation console; questions land as draft; promote to pending.
  3. Phase 2: Contributor role, invite flow, contributor app, review queue, anti-gaming, reliability score job.
  4. Phase 3: Releases (compose, publish, archive); question fetch updates to filter by releaseId.
  5. Phase 4: Stripe integration, gating UI, payouts CSV export.
  6. Phase 5: Reconciliation, spot-audit UI, reliability score surfacing in admin, monitoring.

Feature flags

Each phase ships behind an env-var-driven flag (OFFICIAL_PLAN_PROGRAM_PHASE, integer) that gates route registration and UI rendering. This avoids long-lived feature branches per the project's CONTRIBUTING rules.

13. Open questions

#QuestionDefault if unresolved
Q1 Should rejection comments be visible to other reviewers, or only to admins? Admin-only by default — avoids vote contagion. Revisit if reviewers ask for visibility.
Q2 Is the minimum dwell time per question (5 s) calibrated correctly across question types? Single global threshold for v1. Per-type threshold if data warrants.
Q3 Do we allow a contributor to be both a reviewer and a subscriber on the same exam? Allowed; courtesy examAccess grant with source: 'contributor-courtesy'. Excluded from gaming concerns because they don't pay.
Q4 Should the loader prevent generation against a corpus version that has been edited since last load (hash mismatch)? Hard-fail with explicit message — forces an intentional version bump.
Q5 Stripe Connect for direct contributor payouts vs. CSV export to accounting? CSV export for v1. Revisit when monthly payout volume exceeds ops bandwidth.
Q6 Do we need a per-release pricing model (e.g., one-time purchase of a specific release vs. monthly subscription)? Monthly subscription only for v1. Per-release one-time purchase is a deliberate non-goal.
Q7 Should published questions be re-reviewable (e.g., post-launch corrections feed back into the queue)? No formal flow for v1; corrections go through admin manual edits + flagged-question system. Add a "recall to review" path if needed.
Next step Resolve Q1–Q4 before Phase 0 implementation; Q5–Q7 can be deferred to their respective phases.
